Megha Hegde

10:15 PM

Hi Ziad, thanks for the interesting question. For generative tasks, we need a model with a decoder (so either encoder-decoder or decoder-only). Whether we fine-tune depends on the specific task. For instance, if we just wanted to generate sequences based on a prompt, we would stick to the next-token-prediction pretraining. However, if you wanted to do a specific task like generating promoter variants, you could do supervised fine-tuning on prompt/sequence pairs. I don't have an example of this handy, but I can try to find one or create a simple notebook while you are working on the workshop tasks.

what is the CLS token?

Megha Hegde

11:09 PM

The CLS token is used in BERT to represent a summary of each input sequence. It was introduced to enhance efficiency for tasks where we only need sentence-level predictions. So we can use it as an alternative pooling method. More info here: https://aditya007.medium.com/understanding-the-cls-token-in-bert-a-comprehensive-guide-a62b3b94a941

Fred Gruber

11:11 PM

is there a consensus that using a cls token instead of some sort of pooling is always better?

Megha Hegde

11:12 PM

We generally use mean or max pooling for genomic LLMs rather than the CLS token, as most models in our field aren't optimised to use the CLS token as the best representation of the sequences. However, I believe that in the NLP field, models are more likely to use that CLS token for text understanding

So I would say that for genomic LLMs, use your own pooling
